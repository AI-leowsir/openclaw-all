# Daily Review Template

Use this exact section order.
Keep every section even when content is sparse.

Writing emphasis:
- The main body should prioritize the user's business instructions, newly learned knowledge / newly confirmed facts, and concrete task gaps / deviations / risks.
- Review execution details themselves should be compressed into `复盘范围` and must not dominate the main body.
- Do not let self-explanation about the review process crowd out the user's actual day.

## 1. 今日关键任务
- 用户明确交代的关键任务
- 自主推进且明显影响结果的关键任务

## 2. 今日关键对话
- 提炼关键节点、关键变化、关键确认
- 输入层面必须覆盖所有用户对话
- 输出层面可压缩，但不能省掉关键转折

## 3. 今日关键事实 / 关键结果
- 今天发生了什么
- 形成了哪些关键结果
- 哪些事实最影响结论

## 4. 今日关键偏差
- 偏差是什么
- 严重 / 中等 / 轻微
- 是否属于重复偏差

## 5. 偏差开始节点
- 从哪一刻开始偏
- 哪个判断、哪个转折、哪个动作导致偏离

## 6. 根因分析
- 直接原因
- 根因
- 为什么没有更早发现

## 7. 复盘范围
- 看了哪些会话
- 是否跨会话
- 是否包含群聊
- 是否包含子会话
- 是否包含工具结果
- 是否包含写入/修改过的关键文件
- 复盘动作本身的执行情况仅在这里简要说明，不要在正文前半段展开

## 8. 新增认知候选（可选）
- 只提炼当天新学到的新知识 / 新确认事实
- 供周复盘判断是否进入长期规则 / 长期结论
- 不等于直接更新 `MEMORY.md`
- 不使用机器指令标签

## Writing rules
- Do not reduce the review to result-only summary.
- Do not omit deviation analysis.
- Do not replace evidence with vague impressions.
- If content is limited, write concise text instead of deleting sections.
- Keep the main body centered on the user's actual work/results rather than the mechanics of doing the review.

## 9. 明日改进动作（必填，最多3条）
- 针对今日偏差 / 根因，提出明天具体要做不同的事
- 每条必须：具体、可验证、有时间约束
- 格式示例："明天遇到 X 时，执行 Y 而非 Z"
- 不写空泛的"注意"、"加强"，必须落到可操作的行为层面

## 10. 昨日改进回顾（如有昨日复盘，必填）
- 读取昨日复盘的 `明日改进动作`
- 逐条评估：✅ 已执行 / ❌ 未执行 / ➖ 不适用
- 附简要证据
- 如果昨日复盘没有 `明日改进动作` 模块（旧模板），写"昨日复盘无改进动作，跳过回顾"
