# Deep Review Checklist

Use this checklist before finalizing a review.

## Daily Review Checklist

Confirm all items below:
- Covered key task chains of the day
- Covered key dialogue chains of the day
- Covered key facts and key results of the day
- Identified key deviations
- Identified deviation start points
- Analyzed root causes
- Explained review scope with `复盘范围`
- Verified the review was not based on a single segment
- Verified the review did not collapse into result-only summary
- Verified the body focused on user business instructions / knowledge gain / task gaps rather than review self-explanation
- Verified review-execution details were compressed into `复盘范围`
- If `新增认知候选` was used, verified it contained only same-day new knowledge / newly confirmed facts and did not imply direct `MEMORY.md` update
- Built the daily session index before writing the review
- Verified all candidate sessions / transcripts in scope were enumerated
- Verified all relevant sessions were considered
- Verified daily day-membership was determined after converting transcript timestamps to Asia/Shanghai rather than by raw UTC date strings
- Verified any raw-UTC discovery result was treated only as a temporary hint, not the final inclusion basis
- Verified direct transcripts ignored `(System)` records and only counted `[User]` chats as reviewable evidence
- Verified tool results were considered
- Verified key written/modified files were considered when relevant
- Verified same-day output files / summaries / tool-result chains were checked as backfill evidence rather than ignored after transcript scan
- Verified no same-day output theme remained unexplained by the discovered task chains
- Verified excluded sessions / transcripts were explicitly recorded in the index when applicable

## Weekly Review Checklist

Confirm all items below:
- Summarized the week's major work
- Captured repeated or repeatedly emphasized issues
- Identified key weekly facts
- Identified repeated deviations
- Grouped root causes
- Listed the week's major outputs
- Listed next-week actions
- Evaluated long-term rules / conclusions against the threshold
- If threshold not met, explicitly stated there were no new long-term rules / conclusions
- Built the weekly session index before writing the review
- Verified all candidate sessions / transcripts in scope were enumerated
- Verified direct transcripts ignored `(System)` records and only counted `[User]` chats as reviewable evidence
- Explained review scope with `复盘范围`
- Confirmed whether `MEMORY.md` was updated
- Confirmed the user will be notified after completion regardless of update status
- Verified excluded sessions / transcripts were explicitly recorded in the index when applicable

## Invalid Review Signals

A review is invalid if:
- it only summarizes the latest conversation segment
- it reconstructs no timeline
- it skips deviation analysis
- it writes results without process
- it ignores cross-session evidence when the work was cross-session
- it mixes weekly review content with MEMORY.md storage rules

## Reminder

Do not label the final output as 合格/不合格.
Enforce the standard internally.
