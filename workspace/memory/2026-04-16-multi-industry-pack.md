5. 螺钉、家具、服装这些行业虽然不同，但底层规律一致：  
   **B2B页面最重要的是降低判断成本，不是堆修辞。**

6. 资料可用性边界：
   - `lihongnuo.com-网站分析`：可作为重要样本
   - 个别旧 `.doc`：**只能部分参考，更多依赖图片 / 版面 / 上下文**
   - 螺钉/紧固件材料对“产品页结构化、采购导向”的支撑最强

## B. 分类页可复用规则

### 1）页面目标
分类页不是讲某一个产品。  
它是让采购商快速知道：

- 你有哪些子类
- 它们差别在哪
- 我应该点进哪一类

### 2）推荐结构
建议顺序：

1. 一句话定义：这类产品是什么
2. 分类维度
3. 主要子类 / 产品范围
4. 应用场景
5. 关键选型维度
6. FAQ / 内链 / 轻量信任补充

### 3）可复用写法规则
- 先写“覆盖范围”，再写“差异点”
- 分类维度要服务快速对号入座，不是为了显得专业
- 维度可从这些角度选：
  - 产品类型
  - 材质
  - 结构
  - 规格
  - 用途
  - 工艺
  - 表面处理
  - 适配基材
- 参数只写“选型维度”，不用把具体规格全堆进分类页
- 场景写法要落地，不要大而空

### 4）分类页常见有效区块
- 子类卡片
- 分类入口
- 应用场景列表
- 采购关注点
- FAQ
- 相关产品页 / 栏目页内链

### 5）分类页常见错误
- 写成百科
- 只有行业大词，没有分法
- 子类差异不清
- 套模板只换品名
- 完全没有信任和合作信息

## C. 栏目页可复用规则

### 1）页面目标
栏目页本质上是 **导航级信任页 / 能力页**。  
不是卖单品，而是回答：

- 你们是否值得合作
- 你们能提供什么支持
- 合作怎么推进

### 2）常见栏目类型
- About Us
- OEM / ODM
- Quality Control
- Certifications
- Fabrics / Materials
- Manufacturing Capability
- FAQ
- Case / Solution / Download

### 3）推荐结构
建议顺序：

1. 这页解决什么问题
2. 能力范围 / 服务范围
3. 合作流程 / 支持方式
4. 证明材料（质量、认证、测试、产能、研发、供应链）
5. FAQ
6. 相关页面入口

### 4）可复用写法规则
- 先讲“能解决什么合作问题”，再讲“我们多厉害”
- 用能力说明替代口号
- 用流程、支持项、测试项、产能、定制边界替代空宣传
- 每个模块只解决一个问题
- 适合用：
  - 图文并列
  - 流程图
  - 卡片模块
  - 列表模块

### 5）栏目页常见错误
- 写成纯品牌宣传页
- 只有“专业、优质、经验丰富”
- 没有合作细节
- 把栏目页和分类页混写

## D. 产品页可复用规则

### 1）页面目标
产品页要解决单一产品采购判断。  
核心回答这几个问题：

- 这是什么
- 参数如何
- 适用于什么
- 能不能定制
- 怎么买
- 是否可信

### 2）推荐结构
建议顺序：

1. Product Overview / 一句话定义
2. Key Features（可选）
3. Specifications / 参数
4. Material & Finish
5. Size / Standard / Drawing
6. Applications
7. Customization
8. Packaging / MOQ / Lead Time（有依据才写）
9. FAQ
10. 轻量信任补充

### 3）可复用写法规则
- 少形容，多事实
- 每段只回答一个问题
- 能表格就别硬写长段
- 规格必须独立成区
- 不能所有产品页都一个模板
- 产品差异必须体现在参数重点和模块重点上

### 4）不同品类的重点差异
- **紧固件 / 标准件**  
  参数、标准、尺寸、头型、螺纹、材质、表面处理优先级最高
- **服装 / 内衣**  
  款型、面料、功能、定制项、市场适配、FAQ更重要
- **家具**  
  材质、结构、风格、空间适配、定制、工艺、交付更重要

### 5）产品页常见错误
- 写成科普文章
- 套模板只换产品名
- 为显专业乱堆测试、标准、参数
- 过度营销，AI腔重
- 看完还不知道能不能合作

## E. 跨行业可迁移规则

这些是跨行业基本都能用的：

### 1）底层规则
- 页面先服务判断，再服务SEO
- 先回答问题，再展示品牌
- 好页面是“降低决策成本”的页面
- 模块化、表格化、列表化、图文化，通常优于长段堆字
- 真实、具体、可验证，优于空泛宣传
- FAQ、信任模块、内链结构，多数B2B行业都有效

### 2）结构规则
- **分类页**：定义 → 分类维度 → 场景 → 选型点 → FAQ
- **栏目页**：定位 → 能力 / 流程 → 证明 → FAQ
- **产品页**：定义 → 参数 → 应用 → 定制 → 交易信息 → FAQ

### 3）表达规则
- 先给答案，后给解释
- 一屏一个问题
- 标题即结论
- 用“支持什么 / 覆盖什么 / 适用于什么”替代夸奖式语言
- 合作方式、供货稳定、定制边界、检验支持、交期口径，比修辞更重要

## F. 行业 / 场景特定规则

### 1）服装 / 内衣
- 更常围绕款型、功能、面料、市场定位、人群分类
- 更依赖：
  - 款式差异
  - 面料触感
  - 版型
  - 穿着场景
- 产品页重点常是：
  - 杯型 / 尺码
  - 面料
  - 舒适性
  - 功能点
  - 定制项

### 2）螺钉 / 紧固件 / 标准件
- 必须高度结构化
- 参数、标准、尺寸、材质、表面处理优先级极高
- 适合大量：
  - 表格
  - 规格列表
  - 标准号
  - 图纸支持说明
- 不能写成长文章

### 3）家具
- 更重：
  - 场景图
  - 风格
  - 材质
  - 工艺
  - 配套能力
- 页面更依赖视觉化，而不是纯参数

### 4）站点分析 / 优化建议文档
- 更适合提炼“评价标准”和“问题类型”
- 不适合直接把示例句拿去当定稿模板全站套用

## G. 不能默认套用的点 / 风险点

这里是最关键的边界：

1. **不能默认所有行业都适合长FAQ**

   - 分类页讲范围 / 区分 / 入口

7. **不能默认“更具体”就等于“更好”**
   - 具体必须可验证
   - 不能把竞对常见说法直接当事实写进去

8. **不能默认旧 `.doc` 文本就是完整依据**
   - 若提取差，必须明确：

# 1. 一页总纲

## 核心方法论
**不是把页面写长，而是把页面类型写对。**

三类页面各自职责：

- **分类页**：解决“你这里有哪些，我怎么快速对号入座”
- **栏目页**：解决“你们靠不靠谱，懂不懂行业，能不能合作”
- **产品页**：解决“这一个具体产品能不能买，参数是否匹配，交易是否顺畅”

## 总原则
- 先服务采购判断，再服务 SEO
- 先前置决策信息，再补信任信息
- 页面要帮用户**扫读**
- 模块清楚、结构稳定、标题短、信息可定位
- 少口号、少修辞、少 AI 腔，多事实 / 列表 / 表格 / 图文对应

## 反向学习后得到的判断标准
好页面不是“文案更会写”，而是：

- 信息顺序对
- 信息密度够
- 模块清楚
- 决策信息前置
- 信任信息不抢位

# 2. 三类页面职责表

| 页面类型 | 核心任务 | 用户阶段 | 用户主要问题 | 页面输出结果 | 禁止混用点 |
|---|---|---|---|---|---|
| 分类页 | 做范围说明、分类导航、帮助筛选 | 初筛阶段 | 你们有哪些类型？我该看哪类？ | 用户快速进入正确子类/产品方向 | 不能写成单一产品说明；不能写成行业百科 |
| 栏目页 | 做信任建立、能力解释、合作说明 | 比较评估阶段 | 你们是否靠谱？是否适合合作？ | 用户理解能力边界、合作方式、证明材料 | 不能承担产品决策主任务；不能只做品牌自夸 |
| 产品页 | 做单品采购判断与询盘承接 | 决策阶段 | 这款是否匹配？参数够不够？能否定制？怎么买？ | 用户判断是否询盘 / 打样 / 下单 | 不能写成品牌故事；不能写成泛科普 |

# 3. 三类页面标准模块清单

---

## 3.1 分类页

### 推荐顺序
1. 一句话定义 / 类目说明  
2. 分类维度总览  
3. 主要子类 / 产品范围  
4. 应用场景  
5. 选型维度 / 关键参数维度  
6. 采购常见关注点  
7. 基础信任补充  
8. FAQ  
9. 内链到栏目页 / 产品页  

### 每个模块作用
- **一句话定义**：先告诉用户这页是什么类
- **分类维度总览**：告诉用户可按什么方式区分
- **主要子类 / 产品范围**：做入口分流
- **应用场景**：帮用户判断是否适配
- **选型维度**：告诉用户选这类产品该看什么
- **采购关注点**：承接筛选疑问
- **基础信任补充**：轻量补足合作信心
- **FAQ**：解决高频初筛问题
- **内链**：引导下一步浏览

---

## 3.2 栏目页

### 推荐顺序
1. 页面定位说明  
2. 能力范围 / 服务范围  
3. 合作流程 / 服务流程  
4. 证明模块（质量 / 认证 / 测试 / 产能 / 研发 / 供应链）  
5. 可支持项 / 定制边界  
6. 案例 / 场景支持  
7. FAQ  
8. 内链到分类页 / 产品页 / 联系页  

### 每个模块作用
- **页面定位说明**：明确这页解决什么合作问题
- **能力范围 / 服务范围**：告诉用户你能支持什么
- **流程模块**：降低合作不确定性
- **证明模块**：建立可信度
- **支持项 / 定制边界**：说明能做什么、不能做什么
- **案例 / 场景支持**：增强真实性
- **FAQ**：处理合作层面的阻力问题
- **内链**：导向下一步转化

---

## 3.3 产品页

### 推荐顺序
1. Product Overview / 一句话定义  
2. Key Features（可选）  
3. Specifications / 参数  
4. Material & Finish

6. Applications  
7. Customization Options

### 每个模块作用
- **一话定义**：快速说明这是什么产品
- **核心特点**：提炼结构特点 / 卖点
- **参数**：做第一层采购判断
- **材质与表面处理**：回答材质问题
- **尺寸 / 标准 / 图纸**：回答匹配性问题
- **应用**：说明适用场景
- **定制项**：说明可改到什么程度
- **交易信息**：回答合作可行性
- **FAQ**：解决低频但高阻力问题
- **轻量信任补充**：适度补强，不喧宾夺主

# 4. 三类页面信息优先级表

| 页面类型 | 第一优先级 | 第二优先级 | 第三优先级 | 最后出现 |
|---|---|---|---|---|
| 分类页 | 这类是什么 | 如何区分子类 | 适用于什么场景 / 怎么选 | 轻信任、FAQ、内链 |
| 栏目页 | 这页解决什么合作问题 | 你们能提供什么能力 | 怎么合作、有什么证明 | FAQ、导流入口 |
| 产品页 | 这是什么产品 | 参数 / 标准 / 尺寸 / 材质是否匹配 | 能否定制、怎么买、交期 / MOQ | FAQ、轻信任 |

## 通用优先级原则
- 用户先看：**能不能判断**
- 再看：**能不能合作**
- 最后看：**补充问题**

# 5. 跨行业通用规则清单

## 结构通用
- 分类页：定义 → 分类维度 → 场景 → 选型点 → FAQ
- 栏目页：定位 → 能力 / 流程 → 证明 → FAQ
- 产品页：定义 → 参数 → 应用 → 定制 → 交易信息 → FAQ

## 表达通用
- 先给答案，后给解释
- 标题即结论
- 一屏尽量只解决一个问题
- 用“支持什么 / 覆盖什么 / 适用于什么”替代“专业 / 优质 / 最佳”
- 能列表就别长段
- 能表格就别硬写说明文
- FAQ 短答
- 图文对应，图片承担信息，不只是装饰

## 组织通用
- 参数独立成区
- 应用适合列表 / 卡片
- 栏目页承担信任
- 产品页承担决策
- 分类页承担导航与筛选
- 内链必须服务下一步路径

## 质量判断通用
- 是否让用户更快判断
- 是否把关键决策信息放前面
- 是否减少模板味和 AI 痕迹
- 是否每个模块只做一件事
- 是否真实、具体、可验证

# 8. 后续交给 writing_agent 的输入前提

这部分很重要，后面谁写都得守：

## 1）先确认页面类型
- 必须先判定是：
  - 分类页
  - 栏目页
  - 产品页
- 未判定前不能直接写

## 2）先确认行业
至少确认是否属于：
- 紧固件 / 标准件
- 服装 / 内衣
- 家具
- 其他制造业 B2B

## 3）先确认资料边界
必须先分清：
- 哪些信息已确认
- 哪些只是截图观察
- 哪些旧文档提取不完整
- 资料不足处不能 AI 补全

## 4）先确认可写 / 不可写信息
- **可写**：已确认参数、已确认能力、已确认定制项、已确认交易条件
- **不可乱写**：认证、标准、测试数据、MOQ、交期、付款、性能结论

## 5）必须按页面职责写
- 分类页：只做范围、区分、入口
- 栏目页：只做能力、流程、证明、合作说明
- 产品页：只做单品采购判断与询盘承接

## 6）必须按信息优先级写
- 先判断信息
- 后合作信息
- 再 FAQ 与轻信任
- 不能一上来先讲品牌故事或口号

## 7）必须控制表达方式
- 不写空话、套话、AI 腔
- 不写夸张营销文
- 多模块化、列表化、表格化
- 标题尽量承担判断功能

## 8）必须保留不确定性标记
对提取不完整的旧 `.doc`，必须明确写：
**该文件仅能部分参考，更多依赖图片 / 版面 / 上下文**
